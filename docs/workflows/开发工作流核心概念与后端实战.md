# 开发工作流核心概念与后端实战

> 深入梳理 Claude Code 开发工作流的核心概念，并以后端开发为例演示实际应用。

---

## 一、核心概念梳理

### 1.1 什么是"开发工作流"？

Claude Code 的开发工作流是一套**将 AI 能力组织化、流程化**的方法论。核心思想是：

> **不要把 Claude Code 当作聊天工具，而是当作可编排的工程系统。**

传统方式：你在对话框里一句一句告诉 Claude 要做什么。  
工作流方式：你预先定义好流程模板，Claude 按流程自主执行。

### 1.2 三层架构

Claude Code 的工作流体系是一个**三层架构**：

```
┌─────────────────────────────────────────────┐
│  Command（命令层）                            │
│  用户入口 → 定义流程步骤 → 编排调度            │
├─────────────────────────────────────────────┤
│  Agent（代理层）                              │
│  自主执行者 → 隔离上下文 → 独立工具和权限       │
├─────────────────────────────────────────────┤
│  Skill（技能层）                              │
│  领域知识 → 可预加载 → 可自动发现              │
└─────────────────────────────────────────────┘
```

### 1.3 三层各自的职责

| 层级 | 类比 | 职责 | 文件位置 |
|------|------|------|----------|
| **Command** | 项目经理 | 接收用户需求，拆分任务，分派给 Agent/Skill | `.claude/commands/<name>.md` |
| **Agent** | 专职工程师 | 在隔离环境中自主完成特定任务，拥有独立工具集 | `.claude/agents/<name>.md` |
| **Skill** | 操作手册 | 提供领域知识和操作指南，注入到执行上下文中 | `.claude/skills/<name>/SKILL.md` |

### 1.4 关键概念详解

#### Command（命令）— 流程入口

```yaml
# .claude/commands/my-workflow.md
---
description: "我的工作流"
argument-hint: "[参数]"
---

你是一个工作流编排器。按以下步骤执行：
1. 先调用 Agent(agent: "researcher") 研究需求
2. 再调用 Skill(skill: "code-generator") 生成代码
3. 最后运行测试验证
```

**核心特点**：
- 用户通过 `/my-workflow` 触发
- 运行在**当前对话上下文**中（不隔离）
- 可以调用 Agent 和 Skill
- 适合做**编排入口**

#### Agent（子代理）— 隔离执行者

```yaml
# .claude/agents/db-migrator.md
---
name: db-migrator
description: "数据库迁移专家"
tools: Read, Write, Bash
model: opus
maxTurns: 30
skills:
  - database-conventions    # 预加载数据库规范
memory: project
---

你是数据库迁移专家。负责：
1. 分析现有数据库 schema
2. 生成迁移脚本
3. 验证迁移的前后兼容性
```

**核心特点**：
- 在**全新隔离上下文**中执行
- 有自己的工具白名单、模型、权限
- 可预加载 Skills 作为领域知识
- 支持持久记忆（`memory: project`）
- 适合做**复杂的独立任务**

#### Skill（技能）— 知识注入

```yaml
# .claude/skills/spring-boot-conventions/SKILL.md
---
name: spring-boot-conventions
description: "Spring Boot 项目的编码规范和模式"
---

## Spring Boot 编码规范

### Controller 层
- 使用 @RestController，不用 @Controller
- 统一返回 Result<T> 包装类
- 路径命名用 kebab-case: /api/user-profiles

### Service 层
- 接口 + 实现分离
- 事务注解加在 Service 实现类上
- 复杂查询使用 Specification 或 QueryDSL
...
```

**核心特点**：
- 是**知识/规范**，不是执行者
- 可以被 Agent 预加载（启动时注入）
- 也可以被 Command/用户直接调用
- 适合做**编码规范、API 文档、操作指南**

### 1.5 数据流动方式

```
用户 ──/command──▶ Command
                    │
                    ├── Agent(agent: "xxx")  ──▶ Agent（隔离上下文）
                    │       │                       │
                    │       │ skills: [yyy]          │ 预加载 Skill 知识
                    │       │                       │
                    │       ◀── 返回结果 ────────────┘
                    │
                    ├── Skill(skill: "zzz")  ──▶ Skill（当前上下文）
                    │       │
                    │       ◀── 知识注入
                    │
                    └── 输出最终结果给用户
```

**关键区别**：
- Agent 返回的是**执行结果**（隔离后传回）
- Skill 注入的是**知识上下文**（在当前对话中）

## 二、四种开发工作流对比

| 工作流 | 核心思想 | 适用场景 | 复杂度 |
|--------|---------|---------|--------|
| **RPI** | 研究→规划→实施，三阶段隔离 | 不熟悉的代码库、复杂功能 | ⭐⭐ |
| **Cross-Model** | 多模型互审，减少盲点 | 安全关键代码、复杂重构 | ⭐⭐⭐ |
| **Ralph Wiggum Loop** | 设定目标，自主循环迭代 | 有完善测试的明确任务 | ⭐ |
| **Agent Teams** | 多代理并行，git worktree 隔离 | 大型功能、前后端同步开发 | ⭐⭐⭐⭐ |

## 三、后端实战：用 RPI 工作流开发"用户订单查询 API"

### 场景描述

> 你在一个 Spring Boot + MyBatis-Plus 项目中，需要开发一个**用户订单查询 API**，支持分页、按状态筛选和时间范围过滤。

### 项目结构

```
my-backend/
├── .claude/
│   ├── commands/
│   │   └── backend-feature.md        # 后端功能开发命令
│   ├── agents/
│   │   ├── api-designer.md           # API 设计代理
│   │   └── code-implementer.md       # 代码实现代理
│   └── skills/
│       ├── spring-boot-conventions/
│       │   └── SKILL.md              # Spring Boot 编码规范
│       └── mybatis-plus-patterns/
│       │   └── SKILL.md              # MyBatis-Plus 最佳实践
├── src/main/java/com/example/
│   ├── controller/
│   ├── service/
│   ├── mapper/
│   ├── entity/
│   └── dto/
├── CLAUDE.md
└── pom.xml
```

---

### Step 1：定义 CLAUDE.md（项目记忆）

```markdown
# CLAUDE.md

## 项目信息
- 框架：Spring Boot 3.2 + MyBatis-Plus 3.5
- 数据库：MySQL 8.0
- JDK：17
- 构建工具：Maven

## 编码规范
- Controller 返回统一 Result<T>
- 分页使用 MyBatis-Plus 的 Page<T>
- DTO 使用 record 类型
- 异常使用全局 @RestControllerAdvice 处理

## 常用命令
- 运行测试：mvn test
- 启动服务：mvn spring-boot:run
- 代码检查：mvn checkstyle:check
```

---

### Step 2：定义 Skill（编码规范知识）

```yaml
# .claude/skills/spring-boot-conventions/SKILL.md
---
name: spring-boot-conventions
description: "Spring Boot 后端编码规范，Claude 开发时自动遵循"
---

## Controller 规范
- 类注解：@RestController + @RequestMapping("/api/v1/模块名")
- 方法注解：@GetMapping / @PostMapping，不用 @RequestMapping
- 参数校验：@Valid + @RequestBody（POST）或 @RequestParam（GET）
- 返回类型：统一 Result<T>

## Service 规范
- 定义接口 XxxService，实现类 XxxServiceImpl
- 事务：@Transactional 加在实现类方法上
- 复杂查询：使用 LambdaQueryWrapper

## DTO 规范
- 请求 DTO：XxxQueryRequest / XxxCreateRequest
- 响应 DTO：XxxVO（View Object）
- 使用 Java record 定义

## 分页规范
- 请求参数：pageNum（从1开始）、pageSize（默认10，最大100）
- 响应：PageResult<T>，包含 records、total、pageNum、pageSize

## 异常处理
- 业务异常：throw new BusinessException(ErrorCode.XXX, "描述")
- 全局捕获：@RestControllerAdvice + @ExceptionHandler
```

---

### Step 3：定义 Agent（API 设计代理）

```yaml
# .claude/agents/api-designer.md
---
name: api-designer
description: "设计 RESTful API 接口，输出接口文档和 DTO 定义"
tools: Read, Glob, Grep, Write
model: opus
maxTurns: 15
skills:
  - spring-boot-conventions    # 预加载编码规范
permissionMode: plan           # 只读模式，只设计不实现
---

你是一个 API 设计专家。你的任务是：
1. 阅读需求描述和现有代码结构
2. 设计 RESTful API 接口（路径、方法、参数、响应）
3. 定义请求/响应 DTO
4. 输出设计文档到 docs/api-design.md

注意：
- 遵循预加载的 spring-boot-conventions 规范
- 参考现有 Controller 的风格保持一致
- 考虑向后兼容性
```

---

### Step 4：定义 Agent（代码实现代理）

```yaml
# .claude/agents/code-implementer.md
---
name: code-implementer
description: "根据 API 设计文档实现后端代码"
tools: Read, Write, Edit, Bash, Glob, Grep
model: opus
maxTurns: 40
skills:
  - spring-boot-conventions
  - mybatis-plus-patterns
memory: project
---

你是一个后端开发工程师。你的任务是：
1. 阅读 docs/api-design.md 中的 API 设计
2. 按顺序实现：Entity → Mapper → Service → Controller
3. 编写单元测试
4. 运行 mvn test 验证
5. 运行 mvn checkstyle:check 检查代码风格

实现原则：
- 严格遵循 API 设计文档，不自行添加额外接口
- 遵循预加载的编码规范
- 每实现一层做一次 git commit
```

---

### Step 5：定义 Command（编排入口）

```yaml
# .claude/commands/backend-feature.md
---
description: "后端功能开发完整流程：设计 → 实现 → 测试"
argument-hint: "[功能需求描述]"
---

你是后端功能开发的编排器。按以下流程执行：

## 阶段 1：需求确认
- 阅读用户提供的功能需求：$ARGUMENTS
- 使用 AskUser 工具确认以下信息：
  - 涉及哪些数据库表？
  - 是否需要分页？
  - 是否有权限要求？

## 阶段 2：API 设计
- 调用 Agent(agent: "api-designer") 进行 API 设计
- 将设计文档展示给用户确认

## 阶段 3：代码实现
- 用户确认设计后，调用 Agent(agent: "code-implementer") 实现代码
- 实现完成后展示 git diff 给用户审查

## 阶段 4：验证
- 运行测试：mvn test
- 运行代码检查：mvn checkstyle:check
- 汇总结果给用户
```

### Step 6：实际使用

在终端中启动 Claude Code 并运行：

```bash
cd my-backend
claude

# 运行后端功能开发命令
/backend-feature 开发用户订单查询API，支持按订单状态（待支付/已支付/已取消）筛选，支持按下单时间范围过滤，支持分页
```

#### 执行流程

```
用户输入需求
    │
    ▼
/backend-feature Command 启动
    │
    ├─ 阶段1：AskUser 确认需求
    │   "涉及 t_order 表和 t_user 表"
    │   "需要分页，默认每页10条"
    │   "需要登录，只能查自己的订单"
    │
    ├─ 阶段2：Agent(api-designer) 设计 API
    │   │
    │   │ 预加载 spring-boot-conventions
    │   │
    │   └─ 输出 docs/api-design.md：
    │       GET /api/v1/orders
    │       参数：status, startTime, endTime, pageNum, pageSize
    │       响应：PageResult<OrderVO>
    │
    ├─ 用户确认设计 ✓
    │
    ├─ 阶段3：Agent(code-implementer) 实现代码
    │   │
    │   │ 预加载 spring-boot-conventions + mybatis-plus-patterns
    │   │
    │   ├─ 创建 OrderQueryRequest.java (DTO)
    │   ├─ 创建 OrderVO.java (VO)
    │   ├─ 创建 OrderMapper.java (Mapper)
    │   ├─ 创建 OrderService.java (Service接口)
    │   ├─ 创建 OrderServiceImpl.java (Service实现)
    │   ├─ 创建 OrderController.java (Controller)
    │   ├─ 创建 OrderControllerTest.java (测试)
    │   └─ 每步 git commit
    │
    └─ 阶段4：验证
        ├─ mvn test ✓
        ├─ mvn checkstyle:check ✓
        └─ 汇总报告给用户
```

#### 生成的代码示例

**OrderQueryRequest.java**
```java
public record OrderQueryRequest(
    @Schema(description = "订单状态：UNPAID/PAID/CANCELLED")
    String status,

    @Schema(description = "开始时间")
    @DateTimeFormat(pattern = "yyyy-MM-dd HH:mm:ss")
    LocalDateTime startTime,

    @Schema(description = "结束时间")
    @DateTimeFormat(pattern = "yyyy-MM-dd HH:mm:ss")
    LocalDateTime endTime,

    @Schema(description = "页码，从1开始", defaultValue = "1")
    @Min(1)
    Integer pageNum,

    @Schema(description = "每页条数", defaultValue = "10")
    @Min(1) @Max(100)
    Integer pageSize
) {
    public OrderQueryRequest {
        if (pageNum == null) pageNum = 1;
        if (pageSize == null) pageSize = 10;
    }
}
```

**OrderController.java**
```java
@RestController
@RequestMapping("/api/v1/orders")
@RequiredArgsConstructor
public class OrderController {

    private final OrderService orderService;

    @GetMapping
    public Result<PageResult<OrderVO>> queryOrders(
            @Valid OrderQueryRequest request,
            @AuthenticationPrincipal UserDetails user) {
        PageResult<OrderVO> result = orderService.queryOrders(
            user.getUserId(), request);
        return Result.success(result);
    }
}
```

**OrderServiceImpl.java**
```java
@Service
@RequiredArgsConstructor
public class OrderServiceImpl implements OrderService {

    private final OrderMapper orderMapper;

    @Override
    public PageResult<OrderVO> queryOrders(Long userId,
            OrderQueryRequest request) {
        Page<Order> page = new Page<>(
            request.pageNum(), request.pageSize());

        LambdaQueryWrapper<Order> wrapper = new LambdaQueryWrapper<Order>()
            .eq(Order::getUserId, userId)
            .eq(StringUtils.isNotBlank(request.status()),
                Order::getStatus, request.status())
            .ge(request.startTime() != null,
                Order::getCreateTime, request.startTime())
            .le(request.endTime() != null,
                Order::getCreateTime, request.endTime())
            .orderByDesc(Order::getCreateTime);

        Page<Order> result = orderMapper.selectPage(page, wrapper);

        List<OrderVO> voList = result.getRecords().stream()
            .map(this::toVO)
            .toList();

        return PageResult.of(voList, result.getTotal(),
            request.pageNum(), request.pageSize());
    }

    private OrderVO toVO(Order order) {
        return new OrderVO(
            order.getId(),
            order.getOrderNo(),
            order.getStatus(),
            order.getTotalAmount(),
            order.getCreateTime()
        );
    }
}
```

## 四、总结：为什么要用工作流？

| 直接对话 | 工作流编排 |
|---------|----------|
| 每次手动描述规范 | Skill 自动注入规范 |
| 上下文越来越长，质量下降 | Agent 隔离上下文，始终清洁 |
| 步骤容易遗漏 | Command 预定义完整流程 |
| 无法复用 | 一次定义，团队共享 |
| 每次任务都从零开始 | 记忆系统积累项目知识 |

### 核心原则

1. **Command 做编排，不做执行** — Command 是流程定义，具体工作交给 Agent
2. **Agent 功能特定，不做通用** — 创建 `api-designer` 而非 `backend-engineer`
3. **Skill 注入知识，不注入行为** — Skill 定义"什么是对的"，Agent 决定"怎么做"
4. **CLAUDE.md 保持简洁** — 全局规范放 CLAUDE.md，细分规范放 Skills
5. **每个阶段后压缩** — `/compact` 释放上下文，保持输出质量
